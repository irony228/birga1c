## Биржа заказов 1С

### Краткое описание
Проект — специализированная площадка для заказчиков и программистов 1С. Система помогает быстро найти исполнителя на доработки, консультации или внедрение конфигураций «1С:Предприятие».

Ключевой акцент проекта — прозрачность сделки: от публикации заказа до “финансового” расчёта через внутренний Escrow (холдирование средств).

### Роли и авторизация
- Роли: `Заказчик` и `Исполнитель`
- Авторизация: `email/пароль`, сессия через `JWT`
- Права (RBAC): действия разрешены по роли (кто может создавать/выбирать исполнителя/подтверждать выполнение/оставлять отклики)

### Баланс и тестовые пополнения
- У каждого пользователя есть виртуальный баланс (кошелёк)
- Для тестирования предусмотрено “фиктивное пополнение” (например, `+1000 руб.`) — реальные платёжные шлюзы не используются

## Функциональные требования
### 1. Фронтенд (клиент)
- SPA-интерфейс (desktop + mobile)
- Основные сценарии:
  - регистрация/вход, профиль
  - лента заказов и фильтрация по типу конфигурации 1С
  - создание/редактирование заказа (заголовок, ТЗ, конфигурация, бюджет)
  - карточка заказа: отклики, выбор исполнителя
  - подтверждение выполнения (только для заказчика)
  - уведомления: “колокольчик” + история последних `10–15` событий

### 2. Бэкенд (сервер) — FastAPI
- REST API для:
  - пользователей и ролей + авторизация (JWT)
  - заказов и статусов
  - откликов (`bids`)
  - уведомлений
  - Escrow/транзакций (холдирование и выпуск средств)
- Критичные операции должны быть атомарными:
  - переход статуса заказа
  - холд/релиз денег (чтобы не допустить двойного списания/выпуска)

### 3. Escrow (“Безопасная сделка”) — workflow
Статусы заказа:
- `Открыт` — можно оставлять отклики
- `В работе` — исполнитель выбран, холд активен
- `Завершен` — работа выполнена, средства зачислены исполнителю

Ключевые шаги:
1. `Открыт` → `В работе`
   - Заказчик нажимает `Выбрать исполнителя`
   - система создаёт escrow-транзакцию и замораживает бюджет на балансе заказчика
   - если денег недостаточно — действие не выполняется, просим пополнить баланс
2. Выполнение
   - Заказчик подтверждает выполнение работы
3. `В работе` → `Завершен`
   - система выпускает замороженные средства и зачисляет их исполнителю

Защита от повторной обработки:
- `escrow_transactions` хранит состояние транзакции (например, `frozen` -> `released`) и должен предотвращать повторный релиз

### 4. Уведомления
- События:
  - новый отклик на заказ (для заказчика)
  - исполнитель выбран (для исполнителя)
  - заказ завершен и средства зачислены (для обеих сторон)
- Поле `is_read` для отметки прочтения
- История: последние `10–15` событий в выпадающем меню

## Технический стек (как ориентир)
- Frontend: Nuxt 3 (Vue.js) или аналог, SPA (desktop + mobile)
- Backend: FastAPI, REST API
- Database: MySQL

## Схема данных (минимальный набор сущностей)
- `users` — профиль, роль, текущий баланс
- `orders` — заголовок/ТЗ, конфигурация, бюджет, статус
- `bids` — отклики: цена/сроки, текст, связь с заказом и исполнителем
- `notifications` — текст/тип уведомления, получатель, `is_read`
- `escrow_transactions` — журнал холда/релиза (связь с заказом, сумма, статусы транзакции)

## Тестирование
- Юнит-тесты:
  - проверка стейт-машины статусов заказа
  - логика Escrow (учёт сумм, запреты некорректных переходов)
  - идемпотентность релиза (повторный запрос не выпускает деньги дважды)
- Интеграционные тесты:
  - API под разными ролями (Заказчик/Исполнитель)
  - корректность обновления статусов/транзакций
- e2e (минимум “happy path”):
  - создать заказ → откликнуться → выбрать исполнителя (холд) → подтвердить выполнение (релиз + уведомления)
