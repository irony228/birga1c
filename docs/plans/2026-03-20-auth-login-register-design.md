# Дизайн: страницы `/login` и `/register` (без бэкенда)

## Цель
Добавить в Nuxt 3 интерфейс авторизации:
- `GET /login` — форма входа
- `GET /register` — форма регистрации с выбором роли

На данном этапе бэкенда нет, поэтому сабмит форм — это только демонстрация UX.

## Визуальная структура (Nuxt UI)
Обе страницы используют единый скелет:
- `UPageHero` (заголовок + описание)
- `UCard` (карточка с формой)
- `UFormField` + `UInput` для полей
- `UButton` для отправки

## `/login`
Поля:
- `email` (type `email`)
- `password` (type `password`)

Клиентская валидация:
- email не пустой и соответствует простому regex формата `name@domain.tld`
- пароль не пустой

Поведение при сабмите:
- пока нет бэкенда — показываем inline-сообщение-заглушку
- кнопка блокируется и появляется `loading` на время “симуляции” запроса

Ссылки:
- “Нет аккаунта? Регистрация” -> `/register`

## `/register`
Поля:
- `email`
- `password`
- `confirmPassword` (повтор пароля)
- `role` (радиокнопки): `Заказчик` / `Исполнитель`

Клиентская валидация:
- email: не пустой и корректный формат
- password: не пустой
- confirmPassword: не пустой и совпадает с `password`
- role: не пустая (на UI по умолчанию выбрана `Заказчик`)

Поведение при сабмите:
- пока нет бэкенда — показываем inline-сообщение-заглушку

Ссылки:
- “Уже есть аккаунт? Вход” -> `/login`

## Технические заметки
Реализация выполнена на клиентской валидации и inline-сообщениях.
Когда появится бэкенд, заменим заглушку:
- запросы `POST /auth/login` и `POST /auth/register`
- обработку ошибок и (в зависимости от контракта) сохранение JWT (cookie либо хранение токена в памяти/локальном хранилище).
